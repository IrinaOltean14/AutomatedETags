# AutomatedETags

#### Defining the problem

An **ETag** (Entity Tag) is a standard HTTP response header. It as a unique digital fingerprint or a version identifier for a specific piece of content (a "resource") returned by a web server. Usually, this fingerprint is generated by hashing the contents of the response body using algorithms like MD5 or SHA-256. If the content changes even slightly (like a single character being updated in a JSON response), the hash changes, resulting in a completely new ETag.
They solve two major problems in web development:

- **Wasted Bandwidth:** Preventing the server from sending the exact same large payload to a client that already has it.
- **Slow Load Times:** Allowing the client (like a web browser or mobile app) to confidently load data from its local cache, knowing it is perfectly up to date.

## Documentation

#### Automated ETag Middleware for ASP.NET Core

This is a custom middleware plugin for [ASP.NET](http://ASP.NET) Core Web APIs. It automatically handles HTTP ETags (Entity Tags) so you don't have to write the caching logic yourself for every single endpoint.

It sits in the middle of your server and the internet.

- **For reading data (GET):** It looks at the JSON your server is about to send, creates a unique fingerprint (hash) of that data, and sends it to the user. If the user asks for the data again later, the middleware checks if the fingerprint matches what the user already has. If it matches, it stops the request and just says `304 Not Modified`.
- **For saving data (POST/PUT):** It gives you a helper tool to check if someone else changed the data in the database while your user was looking at it, returning a `412 Precondition Failed` if there is a conflict.

Why use it?

- **Saves Bandwidth:** If the data hasn't changed, it doesn't send the big JSON file again. It just sends an empty 304 response.
- **Stops Data Deletion:** It prevents the problem where two users edit the same page at the same time and accidentally overwrite each other
- **Protects RAM:** It has a built-in safety limit so it won't try to cache massive 5GB video files and crash the server memory.

### Features

- **Smart & Fast Hashing:** It doesn't slow down your app. You can use ultra-fast modern algorithms like `XxHash64` to keep your API lightning fast, or stick to standard ones like `SHA256` and `MD5`.
- **Simple Controller Tags (Attributes):** You don't have to write messy code inside your endpoints. Just put `[EnableETag]`, `[WeakETag]`, or `[SkipETag]` on top of your controllers to control how they behave.
- **Manual Overrides (Semantic Seeds):** Got a clock or a "last generated" timestamp in your JSON that changes every second? You can tell the middleware exactly what data actually matters (like an ID or Version) so the cache doesn't break over useless data.
- **Built-in Observability (Logging):** It prints out debug messages and tells you exactly how many bytes of bandwidth you just saved on every matched request so you can prove it works.
- **Memory Safety (Crash Protection):** It won't accidentally eat all your server's RAM. You can set a `MaxBodySize` limit (like 1MB), and if a payload is too big (like a giant video file), the middleware safely ignores it instead of trying to hash it.
- **Flexible Routing (Global vs. Opt-In):** You are in total control. You can set it to 'Global' to magically cache your entire API by default, or set it to 'Opt-In' if you only want it to touch the specific endpoints you tag.
- **Concurrency Helper (Stop Data Deletion):** Includes an `ETagValidator` tool. If two users try to edit the same data at the exact same time, this tool helps you safely block the second user from accidentally deleting the first user's hard work (returns a `412 Precondition Failed`).
- **Full HTTP Support (Strong & Weak ETags):** It natively understands both types of HTTP caching so you can choose the right tool for the job.
    - You can use **Strong ETags** for things like static files or videos, where you need a perfect, 100% exact byte match or, you can use **Weak ETags** (`W/`) for your dynamic JSON data.

### How to use it

#### Turn it on in `Program.cs`

First, you need to register the tool and tell it how you want it to behave.

**Important:** You must put `app.UseAutomatedETags()` after `app.UseRouting().`

```jsx
using AutomatedETags;

var builder = WebApplication.CreateBuilder(args);

// 1. Configure the rules for your app
builder.Services.AddAutomatedETags(options => 
{
    // OptIn = Only tag specific endpoints. Global = Tag everything automatically.
    options.Mode = ETagMode.OptIn;          
    
    // Choose your hashing alogithm
    options.Algorithm = ETagAlgorithm.XxHash64; 
    
    // Safety limit
    options.MaxBodySize = 1048576;          
    
    // Configure if to use strong or weak ETags
    options.UseWeakETags = false;           
});

var app = builder.Build();

app.UseRouting();

// 2. Turn on the middleware
app.UseAutomatedETags(); 

app.MapControllers();
app.Run();
```

You can customize how the middleware behaves across your entire application. If you don't provide any options, the library will safely fall back to its default settings.

- Mode
    - Default Value: **`Global`**
    - **`Global`**: The middleware runs on every `GET` request automatically. Use `[SkipETag]` to exclude specific endpoints.
    - **`OptIn`**: The middleware stays completely dormant unless a controller or method is explicitly tagged with `[EnableETag]` or `[WeakETag]`.
- Algorithm
    - **`SHA256` / `MD5`**: Standard, universally recognized hashing algorithms.
    - **`XxHash64`**: An ultra-fast, non-cryptographic hashing algorithm. Recommended for high-traffic APIs to minimize CPU usage.
- BodySize (bool): To hash a payload, the middleware must hold it in memory. If an endpoint tries to return a massive payload (like a 50MB video file) that exceeds this byte limit, the middleware will safely skip ETag generation to prevent your server from running out of RAM.
- UseWeakETags (bool): When set to `true`, all automatically generated ETags will be prefixed with `W/` (Weak).

#### Basic Caching (GET)

If you chose `ETagMode.OptIn` in your setup, you need to tell the middleware which endpoints to cache. Just add the attributes to your controller methods.

```jsx
[ApiController]
[Route("api/[controller]")]
public class DataController : ControllerBase
{
    // Use this for exact byte-for-byte matching (Static files, strict JSON)
    [HttpGet("users")]
    [EnableETag] 
    public IActionResult GetUsers() => Ok(database.GetUsers());

    // Use this if want to enable weak ETags
    [HttpGet("stats")]
    [WeakETag] 
    public IActionResult GetStats() => Ok(database.GetStats());

    // Forces the middleware to ignore this endpoint completely
    [HttpGet("logs")]
    [SkipETag]
    public IActionResult GetLogs() => Ok(database.GetLogs());
}
```

Routing attributes:

- `[EnableETag]` : Strong ETag
- `[WeakETag]` : Weak ETag
- `[EnableETag]` : bypass (use this if the middleware mode is set to Global)

#### Advanced Caching (Ignoring Timestamps)

Sometimes your JSON contains data that changes every second (like a `ServerTime` or `LastLogin` field). If you hash the whole body, the cache will break every second.

To fix this, you can give the middleware a **"Custom Seed"** by using CustomETagValue. This tells the library to ignore the JSON body and only hash the specific variables you care about.

```jsx
[HttpGet("profile/{id}")]
[WeakETag] // It's best practice to use WeakETags when using custom seeds
public IActionResult GetProfile(int id)
{
    var user = database.GetUser(id);
    
    // Tell the middleware: "Only hash the ID and the Version number!"
    HttpContext.Items["CustomETagValue"] = $"{user.Id}-{user.Version}";

    return Ok(new {
        user.Id,
        user.Name,
        ServerTime = DateTime.Now // This changes constantly, but the cache still works!
    });
}
```

#### Stopping "Lost Updates" (POST / PUT)

If two users are editing the same page at the exact same time, you don't want User B to accidentally overwrite User A's hard work.

You can use the built-in `ETagValidator` to check the `If-Match` header before you save anything to the database.

```jsx
[HttpPost("update")]
public IActionResult UpdateData([FromBody] string newText)
{
    // 1. Get the current version of the data from the database
    var currentData = database.GetData(); 

    // 2. Check if the user's version matches the database's version
    if (!ETagValidator.IsPreconditionValidWithHash(HttpContext, currentData.Version.ToString()))
    {
        // 412 Precondition Failed: The data changed while the user was typing!
        return StatusCode(StatusCodes.Status412PreconditionFailed, "Conflict: Someone else edited this data.");
    }

    // 3. It's safe! Save the data.
    database.Save(newText);
    return Ok("Saved successfully!");
}
```
